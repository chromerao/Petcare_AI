# Contributing

## 작업 시작

1. GitHub Issue에 문제, 수용 기준, 담당자를 기록합니다.
2. 최신 `main`에서 작업 브랜치를 만듭니다.
3. 가장 가까운 `AGENTS.md`를 읽고 작은 단위로 구현합니다.
4. 테스트와 문서를 함께 수정합니다.

## 브랜치와 커밋

- 브랜치: `feat/123-hybrid-search`, `fix/145-acl-filter`, `docs/88-evaluation-plan`
- 커밋: `feat: add document metadata normalizer`
- 한 커밋에는 한 가지 논리적 변경만 포함하는 것을 권장합니다.

## Pull Request

- PR 본문에 문제, 해결 방법, 검증 결과, 보안/비용 영향을 적습니다.
- 화면 변경은 스크린샷 또는 짧은 영상으로 남깁니다.
- 모델 또는 검색 변경은 동일 평가셋의 전후 결과를 첨부합니다.
- 스키마나 API 변경은 호환성 영향을 명시합니다.

## 리뷰 기준

- 정확성: 요구사항과 실패 경로를 처리하는가?
- 근거성: 인용과 원문이 실제로 일치하는가?
- 보안: ACL 우회, 비밀 노출, 프롬프트 인젝션 위험이 없는가?
- 재현성: lockfile, 설정, 테스트로 재현 가능한가?
- 운영성: 타임아웃, 관측성, 비용 상한이 있는가?
