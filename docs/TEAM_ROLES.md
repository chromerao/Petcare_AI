# 팀 역할 분담

역할은 책임의 빈틈을 막기 위한 기본 오너십입니다. 서로 리뷰하고 페어 작업하되, 최종 의사결정 담당자는 명확히 둡니다.

## 5인 구성

| 역할 | 주 책임 | 주요 산출물 |
|---|---|---|
| 1. PM·도메인·평가 | 문제 정의, 사용자 인터뷰, 백로그, 평가셋, 데모 스토리 | PRD, 질문/정답/근거셋, 주간 지표 |
| 2. 데이터·수집 | 문서 확보, 파서/OCR, 정규화, 메타데이터, ACL | 수집 파이프라인, 데이터 카드, 샘플 문서 |
| 3. RAG·LLM | 청킹, 임베딩, 검색, 재정렬, 프롬프트, 인용 검증 | RAG 파이프라인, 실험 리포트, 회귀 평가 |
| 4. 백엔드·보안 | API, 인증, 작업 큐, 저장소, 오류/비용/로그 정책 | API 계약, 권한 테스트, 운영 대시보드 |
| 5. 프론트엔드·배포 | UX, 인용 뷰어, 피드백 UI, 컨테이너, CI/CD | 웹앱, 배포 URL, 사용성 테스트 |

## 4인 구성

| 팀원 | 결합 역할 | 권장 비중 |
|---|---|---|
| 1 | PM·도메인·평가 + 프론트엔드 UX | 기획 40%, 평가 30%, UI 30% |
| 2 | 데이터·수집 + 저장소 | 수집 60%, 데이터 품질/저장 40% |
| 3 | RAG·LLM + 오프라인 평가 자동화 | RAG 70%, 평가 도구 30% |
| 4 | 백엔드·보안 + 인프라/배포 | API 45%, 보안 25%, 배포 30% |

## 공통 책임

- 모든 팀원은 주 1회 이상 다른 파트 PR을 리뷰합니다.
- 평가셋 작성은 PM만의 일이 아닙니다. 각 파트가 실패 사례를 최소 5개씩 추가합니다.
- 보안과 문서 권한은 백엔드 담당자만의 일이 아닙니다. 수집, 검색, UI에서 모두 검증합니다.
- 데모 전 마지막 주에 기능을 몰아 합치지 않습니다. 매주 배포 가능한 수직 기능을 하나씩 완성합니다.

## 권장 8주 일정

| 주차 | 팀 목표 |
|---|---|
| 1 | 주제/사용자/데이터 확정, 30개 문서 확보, 평가 질문 30개 작성 |
| 2 | 파싱-청킹-검색 최소 흐름, API/UI 연결, 첫 배포 |
| 3 | 메타데이터/ACL, 인용 위치, 기본 검색 평가 |
| 4 | 하이브리드 검색/재정렬 실험, 답변 보류 정책 |
| 5 | 차별화 기능 1개 완성, 프롬프트 인젝션/권한 테스트 |
| 6 | 사용자 피드백, 지연시간/비용 최적화, 관측성 |
| 7 | 회귀 평가, 오류 UX, 운영 문서, 데모 리허설 |
| 8 | 버그 수정, 최종 배포, 발표 자료와 회고 |

## 의사결정 방식

- 되돌리기 쉬운 결정은 담당자가 문서화 후 진행합니다.
- 데이터 이동, 보안, 비용, 핵심 아키텍처처럼 되돌리기 어려운 결정은 팀 리뷰와 ADR이 필요합니다.
- 실험 논쟁은 취향보다 고정 평가셋 결과로 결정합니다.
